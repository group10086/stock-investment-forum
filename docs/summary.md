### 4.2 关键技术实现

| 技术点 | 实现方式 |

| 管理员权限 | require_admin 依赖注入中间件 |
| 禁言机制 | users 表 is_muted + muted_until 字段 |

## 6. 遇到的问题与解决方案

### 6.1 HTTPException 参数错误

**问题**：AI 生成的代码中 HTTPException 使用了 `message=` 参数，但 FastAPI 的 HTTPException 只接受 `detail=` 参数。

**解决**：批量替换 8 个服务文件中约 40 处 `message=` 为 `detail=`。

### 6.2 编码损坏（中文字符 U+FFFD）

**问题**：使用 PowerShell 的 `-replace` 操作修复 HTTPException 时，导致所有服务文件中的中文字符被破坏，UTF-8 多字节字符末尾字节丢失，约 40 处 SyntaxError。

**解决**：编写 Python 修复脚本，精确扫描包含 U+FFFD 的行，根据语境推断并修复原始字符。多次迭代后全部修复。

**教训**：在 Windows PowerShell 中处理含中文的代码文件时，应避免使用 `-replace` 操作，改用 Python 脚本处理。

### 6.3 测试环境数据库连接

**问题**：导入 `main.py` 时触发 `Base.metadata.create_all()` 尝试连接 PostgreSQL 失败，导致测试无法运行。

**解决**：为 `create_all` 添加 `try-except` 保护，允许无数据库时导入。

### 6.4 文档与代码脱节

**问题**：在模块5文档审查中发现，部分文档（backend_api.md、db.md、test.md、user_guid.md）存在与实际代码严重脱节的问题，包括虚构不存在的接口、错误的数据库类型（MySQL 写成 PostgreSQL）、虚报测试用例数量等。

**解决**：基于实际代码逐份重写文档，严格对照路由代码和建表脚本，确保文档内容与代码完全一致。

## 7. AI 协作总结

### 7.1 AI 使用概况

本项目全程使用 AI 辅助开发，涵盖需求分析、架构设计、编码实现、测试调试和文档编写各阶段。AI 使用记录详见 `ai_prompts/ai.md`。

| 阶段 | AI 使用者 | 主要用途 |
|------|---------|---------|
| 模块1 | 邓文博、蔡云飞 | 生成用户故事、交互场景 |
| 模块2 | 邓文博、杨仲晨 | 架构设计、建表脚本 |
| 模块3 | 邓文博、蔡云飞 | 后端代码实现、API 文档 |
| 模块4 | 邓文博 | 单元测试、Bug 修复、联调验证 |
| 模块3-4 | 孟令腾 | 前端组件、管理员页面 |
| 模块5 | 郭森垚 | 文档审查与修复 |

### 7.2 AI 使用的经验教训

1. **AI 生成的代码需要人工审核**：HTTPException 参数错误、编码损坏等问题都是 AI 生成后未经充分审核导致的
2. **AI 生成的文档容易"脑补"**：AI 在编写 API 文档时倾向于根据用户故事补充未实现的功能，必须与实际代码交叉验证
3. **文档质量需要专人把关**：作为文档与 AI 协作负责人，需要建立"AI 生成 → 人工核查 → 与代码对照"的质量流程
4. **提示词需要精确**：越精确的提示词，AI 输出越准确。模糊的提示词容易导致 AI 自由发挥

### 7.3 文档质量红线

在模块5文档修复工作中，确立了以下文档质量标准：

1. **所有接口路径必须与路由代码完全一致**
2. **所有数据库表名和字段必须与 init.sql 完全一致**
3. **技术栈描述必须与实际使用一致**（如 PostgreSQL 不能写成 MySQL）
4. **测试数据必须来自真实运行结果，禁止编造**
5. **所有文档必须标注最后更新日期和维护人**

## 8. 项目不足与改进方向

### 8.1 当前不足

1. **测试覆盖率不足**：47 个 API 接口中仅 5 个有 API 层测试，覆盖率约 10%
2. **无前端测试**：Vue3 组件无单元测试
3. **无性能测试**：未进行压力测试和响应时间基准测试
4. **无安全测试**：未进行 SQL 注入、XSS、CSRF 等安全测试
5. **CI/CD 未完成**：模块2中 CI/CD 配置仅完成 50%
6. **文件上传功能缺失**：attachments 表存在但无上传 API
7. **无密码找回功能**：用户忘记密码无法恢复

### 8.2 改进方向

1. **补充测试用例**：为所有 47 个接口编写 API 层测试，目标覆盖率 80%+
2. **引入 pytest-cov**：量化测试覆盖率，持续改进
3. **添加前端测试**：使用 Vitest 为关键组件编写测试
4. **完善 CI/CD**：配置 GitHub Actions 自动运行测试
5. **补充文件上传**：实现 attachments 表对应的上传 API
6. **添加密码找回**：通过邮件实现密码重置
7. **引入 Redis 缓存**：缓存热点数据，提升性能
8. **引入 Alembic 迁移**：规范数据库版本管理
